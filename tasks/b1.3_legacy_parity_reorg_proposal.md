## 1. Why this exists

After B1.2 shipped, the user compared the new-UI Basic / Appearance / Physics tabs against legacy Win32 screenshots (also captured below for the new session) and found significant divergence in:

- **Section structure** — which fields live in which section, in what order.
- **Field placement across tabs** — several fields are on the wrong tab (rotation fields on Basic instead of Appearance; emit-mode fields on Physics instead of Basic; affected-by-wind on Appearance instead of Physics).
- **Generation mode** — legacy is a tri-state radio mutex (Bursts / Continuous / Weather) but we render it as two independent checkboxes today.

Goal: bring the three tabs back into per-section parity with the legacy editor's resource layout, **without** changing the underlying schema or bridge surface (which already carry every field needed; only the UI mapping changes).

## 3. Proposed structure

### 3.1 Basic tab

| Section | Field | Schema field | Source / move |
|---|---|---|---|
| *(top of tab, no section)* | Name | `name` | ✓ already Basic |
| **Emitter Timing** | Initial spawn delay (seconds) | `initialDelay` | ✓ already Basic |
| | Skip time (seconds) | `skipTime` | ✓ already Basic |
| | Freeze time (seconds) | `freezeTime` | ✓ already Basic |
| **Generation** (tri-state radio mutex) | ◉ Bursts radio | `useBursts === true && !isWeatherParticle` | currently checkbox |
| | ↳ Bursts | `nBursts` | ✓ |
| | ↳ Burst delay (seconds) | `burstDelay` | ✓ |
| | ↳ Particles/burst | `nParticlesPerBurst` | ✓ |
| | ◉ Continuous stream radio | `useBursts === false && !isWeatherParticle` | (implicit) |
| | ↳ Particles/second | `nParticlesPerSecond` | ✓ |
| | ◉ Weather particle radio | `isWeatherParticle === true` | currently Physics |
| | ↳ Particles | **OPEN** (see § 4 Q3) | currently Physics |
| | ↳ Distance from camera (units) | `weatherCubeDistance` | **MOVE** from Physics |
| | ↳ Cube size (units) | `weatherCubeSize` | **MOVE** from Physics |
| | Maximum lifetime (seconds) | `lifetime` | ✓ Basic (label rename) |
| | Minimum lifetime (%) | `randomLifetimePerc` | ✓ Basic (label rename) |
| **Connection** | ☐ Link particles to instance | `linkToSystem` | ✓ Basic (label rename) |
| | Emit mode | `emitFromMesh` | **MOVE** from Physics |
| | Emit offset (units) | `emitFromMeshOffset` | **MOVE** from Physics |

**Tri-state cascade logic** (derived from two flags):
```
mode = isWeatherParticle ? "weather"
     : useBursts        ? "bursts"
     :                    "continuous";

click "Bursts"             → useBursts=true,  isWeatherParticle=false
click "Continuous stream"  → useBursts=false, isWeatherParticle=false
click "Weather particle"   → isWeatherParticle=true (keep useBursts as-is)
```
Sub-fields of inactive radios render disabled (grayed). Always-shown Maximum / Minimum lifetime stays enabled regardless of mode.

**Removed from Basic** (moved to other tabs):
- Random Rotation / Random Rotation Direction / Rotation Average / Rotation Variance → **Appearance > Rotation**
- Parent Link Strength → **Physics > Initial speed** (label: "Parent speed inherit: %")
- Random Scale (`randomScalePerc`) → **Appearance > Textures > Minimum scale** (mapping uncertain — see § 4 Q2)
- Index → **OPEN** (see § 4 Q1)

## 4. Open questions to resolve before drafting the implementation plan

### Q1 — Index field
`index` (schema field; currently on our Basic > Generation) does not appear in any legacy resource. Three options:
- **A: Drop from UI.** Data still in schema; just no edit affordance. Risk: users can't edit it at all.
- **B: Add a small "Atlas" or "Texture" section** somewhere (likely Appearance > Textures) to host Index alongside texture-related fields.
- **C: Keep at current Basic > Generation.** Diverges from legacy.

Recommendation: **A or B**. The user's call.

### Q2 — `textureSize` / `nTriangles` / `randomScalePerc` mapping
Legacy Appearance > Textures has two spinners (IDC_SPINNER2 "Texture elements", IDC_SPINNER13 "Minimum scale %"). Our schema has three plausibly-related fields. Definitive mapping requires reading `src/UI/Emitter.cpp`'s `WM_INITDIALOG` to see which spinner ID is wired to which model field. **Action for the next session**: grep `Emitter.cpp` for `IDC_SPINNER2`, `IDC_SPINNER13`, and resolve each.

Current best guesses (NOT verified):
- `textureSize` ↔ "Texture elements" (high confidence)
- `randomScalePerc` ↔ "Minimum scale %" (medium confidence)
- `nTriangles` ↔ (no legacy label — orphan; may need its own section or get dropped)

### Q3 — Weather "Particles" mapping
Legacy Basic > Generation > Weather branch has a "Particles:" spinner (IDC_SPINNER35). Schema weather fields are: `weatherCubeSize`, `weatherCubeDistance`, `weatherFadeoutDistance`. The first two clearly match "Cube size" + "Distance from camera". The "Particles" mapping is uncertain — possibilities:
- `weatherFadeoutDistance` IS the legacy "Particles" field (badly named in the schema)
- "Particles" reuses `nParticlesPerSecond` (same schema field, different label per mode)
- There's a 4th weather schema field this proposal is missing

**Action for the next session**: grep `Emitter.cpp` for `IDC_SPINNER35` to resolve.

### Q4 — `groups[1]` (Lifetime random-param)
Our schema's `groups` array has 3 entries: `[0]` = Initial Speed, `[1]` = Lifetime, `[2]` = Initial Position. Legacy Physics tab only surfaces `[0]` + `[2]`. Three options:
- **A: Hide from UI** entirely. Risk: any saved data using it becomes unreachable.
- **B: Surface elsewhere** — possibly as a "Lifetime variance" sub-block inside Basic > Generation.
- **C: Keep visible on Physics** as an extension section, explicitly out-of-spec for legacy parity.

Recommendation: **A unless we discover it's actively used by data the engine reads**. The user's call.

### Q5 — Trailing colons on labels
Legacy uses ":" after every label ("Initial spawn delay:", "Skip time:", "Freeze time:"). Our current labels don't ("Initial Delay", "Skip Time"). Style decision:
- **Match legacy**: add ":" to every field label.
- **Keep colon-less convention** (from the 2026 design source).

Cosmetic only. The user's call.

## 5. Estimated dispatch shape

If approved, B1.3 would be a single dispatch (~8–10 commits):

1. **Basic tab restructure** — heaviest commit. Generation tri-state mutex requires the most thought (derived state, three explicit click handlers, cascading enabled-ness). Move emit-mode fields in; move rotation fields out.
2. **Appearance tab restructure** — wrap into 5 Sections. Label renames. Move rotation fields in; move affected-by-wind out.
3. **Physics tab restructure** — surface `groups[2]` (Initial position) and `groups[0]` (Initial speed) as top sections. Wrap remaining fields into Acceleration + Ground interaction. Move emit fields out; move weather fields out; move parent link strength in; move affected-by-wind in.
4. **Label-rename pass** (~10 labels) — small commit.
5. **Test rewrites** — many existing specs assert on label text or DOM structure; expect ~30-50 specs to need updates.
6. **Docs** — CHANGELOG + HANDOFF.

No bridge schema changes. No C++ work (unless Q3 or Q2 surfaces a missing schema field — unlikely).
