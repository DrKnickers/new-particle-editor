# New-UI audit — 2026-06-09

_Multi-agent workflow: 7 audit lenses → per-finding adversarial verify → settings brainstorm → synthesis. 56 agents, 3.16M tokens. **47 findings, 24 confirmed + 23 adjusted, 0 refuted.**_


## Executive summary

This is a mature, well-architected arch-C editor: the two highest-frequency bridge channels (stats/tick, viewport/frame-ready) are correctly isolated to leaf components, so the audit's blast radius is bounded (confirmed by emitter-tree/StatusBar isolation finding). The dominant theme is a *per-edit IPC + re-render storm*: a single scalar property edit fans out into a full EngineStateDto serialize + full emitter-tree walk + three redundant re-fetch round-trips host-side, while the React tree re-renders un-memoized rows/tabs/SVG on every scrub pixel because the editor uses React.memo zero times. The cheapest, highest-leverage wins are two surgical host-side guards (only emit tree/changed when a tree-visible field changed; consume the tree/changed payload instead of re-requesting it) plus rAF-coalescing the spinner/pointermove commit paths. The second theme is *missing safety/feedback*: emitter delete is unconfirmed from four entry points, and Save/Open/Import/autosave-recover failures are silently swallowed with no toast system anywhere. The third is *accessibility structure*: the emitter tree exposes no hierarchy ARIA, nests an interactive eye-toggle inside a row <button>, and the curve canvas has no keyboard selection entry point. UX-consistency findings are mostly low-risk token cleanups (accent-sky-500 bypass, dead primary-button hover, amber warning boxes). Several bundle findings were downgraded once verifiers confirmed this is a local-disk WebView2 load, not a network SPA — the real cost is parse/compile, not download bytes. Settings work centers on a delete-confirm toggle and exposing already-persisted state (autosave cadence, default dock, layout reset) through the near-empty Preferences dialog.

### Why-now detail


**1. Only emit emitters/tree/changed when a tree-visible field changed in set-properties** _(performance, S/high)_

Every scalar edit (lifetime/gravity/blendMode) unconditionally calls both EmitEngineStateChanged AND EmitEmittersTreeChanged, which walks the whole tree and fans out into three redundant re-fetch round-trips, even though the tree DTO only carries id/name/role/linkGroup/visible/children. A one-line guard (treeVisibleChanged = patch.contains("name")) removes the tree walk plus two of three round-trips per edit. Highest payoff-to-effort in the report.


**2. Consume the tree/changed payload instead of re-requesting emitters/list** _(performance, S/high)_

EmitEmittersTreeChanged already ships the complete {root: tree} (byte-identical to emitters/list), but EmitterTree's handler discards e.payload and fires a fresh emitters/list request, re-walking and re-serializing the identical tree plus a full round-trip on every mutation. Two-line fix: bridge.on("emitters/tree/changed", e => setTree(e.payload)). Compounds with rank 1.


**3. Surface Save / Open / Save As failures (silent data-loss risk)** _(ux, M/high)_

file/save|open|save-as return {ok:false; error} but every Toolbar/MenuBar/Ctrl+S call site fires void bridge.request and discards the result. A failed save (disk full, read-only, locked) leaves the doc silently dirty while the user believes it saved — real, unaware data loss. Host already separates user-cancel from IO failure; route all four call sites through a shared helper that pops an error Modal on the non-cancel !r.ok case. Pure web-side; no host change.


**4. rAF-coalesce spinner scrub + pointermove commits** _(performance, M/high)_

Spinner fires onChange on every drag pixel / wheel notch / 50ms hold-repeat, and onPointerMove sends a full viewport/input round-trip per native mouse poll (125-1000 Hz). Neither path debounces. Coalescing both to once-per-rAF caps the engine's per-move unproject and the snapshot+tree serialize storm to display-refresh rate. Largest combined win when paired with ranks 1-2. Magnitude needs a profiler on a running arch-C host.


**5. Confirm before deleting emitters (and subtree)** _(usability, M/high)_

Delete is the most destructive structural action (removes the emitter and its whole child subtree) yet fires immediately from four no-friction entry points including a single Delete keypress when the tree holds focus — while the app DOES gate cheaper actions (Reset View Settings pops a confirm). Reuse the existing Modal/AlertDialog pattern; gate non-trivial/subtree deletes. Doubles as the highest-value Settings toggle.


**6. Memoize EmitterRow + per-row selection subscription to stop full-tree re-render on marquee/drag** _(performance, M/high)_

EmitterRow is un-memoized; a marquee sweep calls setIds (no equality guard) on every pointermove, re-invoking every row each frame. Wrap in React.memo and switch to the already-exported useIsEmitterSelected(node.id) per-row subscription so only rows whose membership flips re-render. Single highest-frequency render hotspot in the tree; pairs with ranks 12/13.


**7. Fix accent-sky-500 token bypass on checkboxes/radios/sliders** _(ux, S/medium)_

base.css globally sets accent-color: var(--accent) and documents the intent, but 9 controls hardcode accent-sky-500 (#0ea5e9), a theme-static wrong-shade blue that doesn't flip with the light theme. Delete the class at 8 checkbox/radio sites; replace with accent-[var(--accent)] at ColorButton.tsx:237 (a range input base.css doesn't match). Trivial, removes a documented-contract violation.


**8. Add Enter-to-submit + initial focus to Spinner modals and default-focus Save in SaveChangesPrompt** _(usability, S/high)_

Rescale/RescaleEmitter/IncrementIndex never focus a field on open and never wire Enter to OK; worse, Spinner's Enter just blurs, so a user who types a value and presses Enter sees nothing happen. SaveChangesPrompt's most consequential modal defaults focus to Cancel, so Enter risks cancelling. ModNicknameDialog already proves the correct pattern. Low-effort, fixes a daily friction.


**9. Add emitter-tree hierarchy ARIA (aria-level / setsize / posinset)** _(accessibility, M/high)_

The tree is a flat <ul role=tree> of <li role=treeitem> with depth conveyed only by paddingLeft, so a screen reader announces every root/lifetime/death child as a level-1 sibling — the parent->child spawn relationship that is the tree's entire purpose is invisible to AT. depth and sibling counts are already on the FlatRow. Highest a11y payoff for moderate effort.


**10. Show open filename + dirty indicator in-app** _(usability, S/medium)_

The file path and dirty flag are tracked in useFileState() but presented ONLY in document.title; under arch-C the React app is the chrome and the OS title bar may be minimal/absent. The user cannot answer 'which file?' / 'unsaved?' from the editor surface. Presentation-only add to the header or StatusBar. Whether the title-bar gap actually bites depends on host frame styling.


**11. Fix nested interactive eye-toggle inside row <button> + apply roving tabindex** _(accessibility, M/medium)_

Each row is a <button> containing a <span role=button tabIndex=0> visibility toggle — invalid interactive nesting with undefined activation, and it doubles the tab stops (2N+ for N rows, no roving tabindex). Make the eye tabIndex=-1 / a sibling button and give only the primary row tabIndex=0 (arrow keys already navigate). Fixes the markup defect and collapses tab order to one stop.


**12. Move per-row leaveLinkGroupDisabled subtree walk + clipboard subscription out of the hot render path** _(performance, S/medium)_

Two cheap allocation removals that ride the marquee re-render storm: leaveLinkGroupDisabled rebuilds a Map and walks the row subtree on every render purely to disable a context-menu item only visible when the menu is OPEN (compute it lazily in onOpenChange); and each row opens its own clipboard store subscription (lift to one tree-level subscription). Both are S-effort and compound with rank 6.


**13. Lazy-load on-demand modal dialogs and the demo routes out of the startup chunk** _(performance, M/medium)_

Eight closed-on-first-paint dialogs (LinkGroupSettings 461 LOC, SetLinkGroup, ImportEmitters, About, Rescale*, IncrementIndex) plus three ?demo= routes are statically imported into App.tsx, so their code parses/compiles at startup despite being unreachable until a click. Convert the boolean/atom-gated ones to lazy()+Suspense fallback={null}. Note: AutosaveRecoveryDialog also exports a view used synchronously by the demo route, so split it last. Local-disk WebView2 — win is parse time, not download.


**14. rAF-throttle curve-key live-drag so the SVG doesn't re-render per pointermove** _(performance, M/medium)_

handleKeyDragMove calls setLiveDrag every pointermove, re-rendering the 1614-LOC CurveEditorPanel + un-memoized CurveEditor SVG — but the visual drag already works off the inner dragRef, so setLiveDrag exists almost solely to update the single-key spinner readout. Route the readout through a ref/tiny subscriber (or rAF-throttle); React.memo alone won't help because the inner component self-renders via setDragTick. Redundant second render pass, hence medium.


**15. Fix dead primary-button hover + amber warning-box palette (token-based)** _(ux, M/medium)_

Two correlated theme-token cleanups. Primary CTAs use hover:bg-accent (same colour as base) = zero hover feedback, while --accent-2 is reserved for exactly this and used nowhere; swap to hover:bg-accent-2 on Modal.OkButton + 4 copies. Separately, link-group warning boxes hardcode bg-amber-200/text-amber-900 — an opaque light card inside dark chrome — instead of the existing --warning/--danger tokens; replace with bg-warning/15 + extract a shared ConflictNote (byte-identical duplication across two dialogs).


## By dimension


**performance** (14 confirmed) — Dominant theme is the per-edit IPC + re-render storm. Two surgical host-side fixes lead: (1) set-properties unconditionally emits both engine/state/changed AND emitters/tree/changed, triggering a full tree walk + three re-fetch round-trips for any scalar edit (BridgeDispatcher.cpp:3099-3104); (2) tree/changed ships the full tree but every subscriber discards it and re-requests emitters/list (EmitterTree.tsx:1179-1181). On the web side, the editor uses React.memo zero times, so un-memoized EmitterRow re-renders on every marquee/drag pointermove (no equality guard on setIds), EmitterPropertyTabs re-renders all ~18 fields per scrub pixel, and the curve SVG re-renders per key-drag frame. The common multiplier is uncoalesced commit paths: Spinner fires per-pixel/per-notch/per-50ms and pointermove sends a full round-trip per native mouse poll; rAF-coalescing caps both to refresh rate. Bundle findings (single 596KB chunk, eager dialogs, demo routes, mock bridge, font-display:block) were mostly downgraded because this is a local-disk WebView2 load, not a network SPA — the real cost is parse/compile, and several should be host-profiled before investing. One finding is a positive confirmation that stats/tick and frame-ready are correctly isolated, bounding the blast radius. Every magnitude claim needs a profiler on a running arch-C host; the code-level costs are confirmed.


**usability** (7 confirmed) — Two clusters. Safety/feedback: emitter delete (the most destructive structural action, removing whole subtrees) fires with no confirmation from four entry points while the app gates cheaper actions; the empty emitter tree shows only a terse '(no emitters)' with no call to action. Keyboard/focus: Spinner-based modals lack Enter-to-submit and initial focus (and Spinner's Enter just blurs), SaveChangesPrompt defaults focus to Cancel; tree keyboard nav can't extend a multi-selection (no Shift+Arrow) though batch ops need one; Edit-menu Ctrl+C/X/V/Del hints imply global shortcuts that only fire with tree focus. State visibility: open filename + dirty flag live only in document.title (weak under arch-C chrome), and View/Emitters menu dock items never reflect open state though the toolbar does. Most are S-effort and reuse existing patterns/data.


**ux** (8 confirmed) — Mostly low-risk design-token consistency cleanups plus two feedback gaps. The sharpest functional gap is silent Save/Open/Save-As failure (no toast system exists anywhere), a real data-loss risk; Import success and autosave-recovery also swallow their outcomes (the autosave-recover fix is cross-layer because the host already DeleteOrphans on failure and returns no status field). The cosmetic set: accent-sky-500 hardcoded over the theme accent token on 9 controls; primary CTA hover is a no-op (hover:bg-accent == base) while the reserved --accent-2 is unused; amber/red warning boxes use raw Tailwind palette instead of --warning/--danger tokens, rendering as opaque light cards in dark chrome; Modal vs ToolPanel body-surface divergence (downgraded to low — only a thin gutter shows it); four different input-surface tokens across dialogs; and dialog title casing drift (Title Case vs sentence case). The token fixes are individually S-effort and collectively tighten the design-system contract.


**accessibility** (8 confirmed) — Structural gaps in the two core surfaces. Emitter tree: no hierarchy ARIA (flat treeitems, depth only visual — spawn relationship invisible to AT), and an invalid interactive nesting (eye-toggle <span role=button> inside the row <button>) that also bloats tab order to 2N+ stops with no roving tabindex. Curve editor: the canvas is role=img with no keyboard SELECTION entry point for an on-canvas key (the verifier corrected the original 'entirely mouse-only' framing — Delete, copy/cut/paste, and arrow-key time/value nudging of an ALREADY-selected key already work via the panel + Spinner; only the selection/insert entry point and the role=img mislabel are the genuine gap, so severity dropped to medium). Token contrast: --text-3 fails WCAG AA (~3.8:1 dark, ~3.34:1 light) on always-on small-print labels, not just disabled text. Smaller items: link-group membership absent from row accessible names, custom color-slot clear is right-click-only with no keyboard path, and the Win32-UIA golden specs auto-skip in the shipped composition default (coverage asymmetry, partially mitigated by the existing composition-reachable spec). Several fixes will require updating a11y golden snapshots.


**maintainability** (2 confirmed) — Two structure-only seams flagged under the perf/ux lenses. renderBody recreates BasicTab/AppearanceTab/PhysicsTab inline with fresh closures and no React.memo barrier (EmitterPropertyTabs.tsx:239) — the verifier corrected the framing: a memo barrier would NOT fix the scrub scenario (setProperties allocates a new object every frame, defeating it), so the honest value is a maintainability seam enabling future per-field memoization, not a scrub-perf win. Separately, modal footer buttons are hand-rolled in 3 dialogs (SaveChangesPrompt, AutosaveRecoveryDialog, MenuBar reset-view) instead of reusing Modal.CancelButton/OkButton, and the copies drift by omitting disabled:cursor-not-allowed/opacity-50; extract a shared <ModalButton variant> so all footer buttons (and the dead-hover fix) live in one place. Both are M-effort and reduce future change surface.

## Settings — full proposals


_Current state: The Preferences dialog (web/apps/editor/src/screens/PreferencesDialog.tsx, 42 LOC) currently exposes exactly ONE control: a 3-way Theme toggle (Dark / Light / System), persisted to localStorage key `alo:theme` via lib/theme.ts (readStoredMode/applyMode at theme.ts:9-26). No other settings live in the dialog.

Other persisted state DOES exist, but is scattered and NOT user-configurable from Preferences:
- Panel layout sizes — localStorage `alo:layout:*` (PanelLayout.tsx:78-134); resettable only via View menu.
- Right-dock target — localStorage `alo:right-dock`, default "spawner" (lib/right-dock.ts:13,29-40).
- Custom-color palette — localStorage via primitives/palette-store.ts; native-host registry-backed in arch-C.
- Last-used mod — registry `HKCU\Software\AloParticleEditor\LastMod`, written by `mods/select` (bridge-schema index.ts:893-895); restored at next launch host-side.
- Recent files — registry-backed, capped at kMaxRecentFiles=9 host-side (BridgeDispatcher.cpp:211).
- Lighting Force-Align flag — registry `LightingForceFillAlignment` via `settings/lighting-force-align/set`.
- Autosave — runs host-side on hardcoded 30s/5min timers (Autosave.h:41-44), always on; no enable/interval control. Web side only does crash-recovery on mount (App.tsx:202).

Reduced-motion is honored from the OS only (PanelLayout.tsx:236-238); there is no in-app override. Delete is unconditional with no confirmation (EmitterTree.tsx:411-413, 975-977, 1673-1683). The established pattern for a new app-level toggle is a tiny localStorage-backed read/apply module exactly like theme.ts; the dialog uses Modal.tsx and primitive radio/toggle styling already present._


### Confirm before deleting emitters _(high, M)_
- **Controls:** A toggle that gates emitter deletion behind a Radix AlertDialog confirmation. When on (default), deleting one or more emitters (Delete key, toolbar trash, context-menu Delete) shows a 'Delete N emitter(s)?' prompt; when off, delete is immediate as it is today.
- **Why:** Emitter delete is currently UNCONDITIONAL from three entry points and the Delete key fires on a focused tree row (EmitterTree.tsx:411-413 context menu, :975-977 toolbar, :1673-1683 keyboard). A misplaced keystroke silently destroys an emitter and its whole subtree. Undo exists, but a particle artist rapidly editing a tree can lose work without noticing which emitter vanished. This is the single highest-value safety setting for THIS workflow.
- **Default:** on (confirm)
- **Feasibility:** All three delete handlers route through `bridge.request({kind:'emitters/delete'})`. A localStorage-backed flag (mirror theme.ts) read inside each handler, plus reusing the existing AlertDialog pattern already used by Reset View Settings (engine/action/reset-view-settings, MenuBar.tsx) and SetLinkGroupDialog. Hook points: EmitterTree.tsx:411, :975, :1673. The multi-select count is already available via useEmitterSelectionStore (EmitterTree.tsx imports it).

### Autosave interval (or Off) _(high, L)_
- **Controls:** A select for the recent-tier autosave cadence — e.g. Off / 15s / 30s (default) / 60s / 2min — controlling how often the host writes the crash-recovery snapshot. Optionally a second control for the stable-tier (5min default).
- **Why:** Autosave runs on HARDCODED 30s/5min timers that are always on (Autosave.h:41-44, wired at HostWindow.cpp:2116-2117). Each tick serialises the whole ParticleSystem to %TEMP% and is dirty-gated, but on very large systems the 30s write can cause a perceptible hitch, and some users editing throwaway test files want it off entirely. Letting the user pick the cadence (or disable) respects both the safety-conscious and the performance-conscious workflows.
- **Default:** 30s recent / 5min stable (current)
- **Feasibility:** Needs a new bridge message (e.g. `settings/autosave` { recentMs; stableMs } or { enabled; intervalSec }) so the host can KillTimer/SetTimer with the chosen interval and persist to HKCU like LastMod does. The autosave timer plumbing already exists (HostWindow.cpp:2107-2143); only a setter + registry read at startup are new. Web side: a localStorage cache for instant UI + a bridge push. needs host verification that re-arming the timer mid-session is glitch-free.

### Restore last mod on startup _(medium, M)_
- **Controls:** A toggle: when on (default), the editor re-selects the mod that was active when it last closed; when off, it always starts Unmodded. Optionally a dropdown to pin a SPECIFIC default mod instead of 'last used'.
- **Why:** Mod selection already persists to HKCU\...\LastMod on every `mods/select` (bridge-schema index.ts:893-895) and is restored at launch host-side. But that behavior is invisible and non-optional. A modder who juggles several mods may want a fixed default; another may want to always start clean. The whole capability (persist + restore + the mod list) already exists — this just exposes the policy.
- **Default:** Restore last mod (on)
- **Feasibility:** The mod list + active path come from `mods/list` (returns ModDescriptor[] + activePath). The 'pin a specific mod' variant reuses `mods/select`. A localStorage flag gates whether the host's existing restore runs, OR a new registry value chosen via the dialog. Hook: the host LastMod restore path (BridgeDispatcher mod handlers) + a small Preferences section calling `mods/list`. needs host verification for the registry-policy wiring.

### Default right-dock panel _(medium, S)_
- **Controls:** A 3-way choice for which side panel is open on a fresh launch: Spawner (current default) / Lighting / None.
- **Why:** The right dock holds either Spawner or Lighting or nothing, and already persists per-user to localStorage `alo:right-dock` (lib/right-dock.ts). But the persisted value IS the last-session state — there's no way to say 'always start with Lighting' or 'always start closed'. Lighting-focused users currently have to reopen Lighting every launch. Tiny surface, real daily friction.
- **Default:** Spawner
- **Feasibility:** right-dock.ts already centralises read/persist (readInitial at :29, persist at :42). Add a second localStorage key `alo:right-dock-default` and have readInitial prefer it when set. The Preferences control writes that key. Fully web-side, no bridge work. Hook: lib/right-dock.ts:29-40.

### Reduce motion (panel slide animations) _(medium, M)_
- **Controls:** A 3-way override — Follow OS (default) / Always animate / Never animate — for the dock-slide and other UI transitions, independent of the OS prefers-reduced-motion setting.
- **Why:** Motion is currently driven SOLELY by the OS media query (PanelLayout.tsx:236-238). Users on a machine where they can't or won't change the OS setting have no in-app escape hatch. The dock-slide drives a host-side interpolated viewport re-render (animate-scene-rect) every frame during the tween; a user who finds it distracting (or who hits stutter on a weak GPU) currently cannot turn it off without changing Windows settings. An explicit app override is the standard accessibility affordance.
- **Default:** Follow OS
- **Feasibility:** Wrap the `window.matchMedia('(prefers-reduced-motion: reduce)')` check (PanelLayout.tsx:236-238) in a helper that first consults a localStorage override. The CSS @media reduced-motion blocks (components.css:128,583,1113,1271) would also need a `[data-reduce-motion]` html-attribute companion (set like data-theme) to cover the pure-CSS transitions. Hook: PanelLayout.tsx:236, plus a data-attribute applied at root like applyMode does.

### Disable frosted-glass modal backdrop _(medium, M)_
- **Controls:** A toggle (default off, i.e. blur on) that, when enabled, drops the modal's engine-snapshot capture + backdrop-blur in favor of a plain dimmed overlay.
- **Why:** Opening any Modal triggers a `viewport/capture-snapshot` GPU readback + PNG encode + base64 IPC + decode, which the code itself documents as ~50-500ms at large resolutions (Modal.tsx:158-165), plus a `backdrop-blur-sm` filter (Modal.tsx:147-148). On a weak GPU or a 4K viewport this makes every dialog feel sluggish. A 'plain backdrop' escape hatch removes the most expensive per-modal-open cost for performance-sensitive users. HYPOTHESIS grounded in the documented capture cost; needs host verification for the actual frame-time saved.
- **Default:** Frosted glass on (toggle off)
- **Feasibility:** Modal.tsx already branches on whether a snapshot is ready (Modal.tsx:169-179 capture path).
