## 1. Goal + scope

**Goal.** Replace the editor's current "pick a background texture" model ([`BackgroundPicker.tsx`](web/apps/editor/src/screens/BackgroundPicker.tsx:77)) with reading the **actual skydome(s)** the game and/or the active mod defines, and rendering them behind the preview, so the background matches the in-game environment. The Alamo engine exposes a **primary** and a **secondary** skydome (community parlance "Dome1 = starfield, Dome2 = backdrop"); the user should be able to select **both**, mirroring the Petroglyph map editor's two skydome pickers. (Whether the two slots are *independently* pickable or the pair is constrained by the GameObject is itself unconfirmed ‚ÄĒ ¬ß5a, fork b/e.)

**In:**
- A reusable **map-environment reading layer** (`SkydomeEnvironment` / `MapEnvironment`, ¬ß3) that resolves, for the active game/mod, which skydome asset(s) are referenced and where they live. (Built reusably on purpose ‚ÄĒ see the LT-8 coupling note below; concrete struct + loader signature proposed in ¬ß3.)
- Loading + rendering the real skydome behind the scene through the existing `FileManager` mod‚Üíbase‚ÜíMEG resolution chain ([`src/managers.cpp:13`](src/managers.cpp:13)).
- Support for **primary + secondary** skydome selection (two active slots / two domes), not the current single-active-slot model.
- A dual-slot picker UI in the new-UI, living where the current `BackgroundDropdown` sits ([`Toolbar.tsx:229`](web/apps/editor/src/components/Toolbar.tsx:229)).
- Persistence of the dual-slot selection that survives a restart in the **new-UI** daily-driver (closing the current new-UI persistence gap ‚ÄĒ see ¬ß4.4).

**Out (with reasons):**
- **Full 1:1 parity with the game's `Skydome.fx`** (clouds + SPH lighting + atmosphere). Deferred to the "recreate game pipeline" track (LT-8 depends on MT-15; MT-16/LT-5 render-triage already flags scene colour-grade as game-pipeline work ‚ÄĒ MEMORY `project_mt16_lt5_render_triage`). MT-15 should **lay data plumbing LT-8 consumes**, not ship a full forked dome shader. Phase split (¬ß3) keeps fidelity scope-bounded. **Note:** "no full parity" still leaves two parity sub-decisions that fork (a) must make explicitly, not implicitly drop: **cloud animation** (the real shader scrolls clouds via a time uniform) and **SH per-vertex lighting** ‚ÄĒ both addressed under fork (a) below rather than silently omitted.
- **Cloud animation (scrolling clouds), unless fork (a) explicitly opts in.** The real shader scrolls clouds via `CloudScrollRate` driven by a time uniform (¬ß2.4); the editor's dome path feeds **no** time value today (¬ß2.1 / fork a). Default scope is a **static** cloud overlay (or no cloud overlay); animated clouds are a deliberate opt-in costing a new time handle wired through `InitSkydomeEffect` + `RenderSkydome` (fork a sub-fork). If not opted in, scrolling clouds are **Out**.
- **Legacy (`--legacy`/arch-A) dual-slot UI.** Legacy is greenlit for removal (MT-13, MEMORY `project_daily_driving_legacy`); the legacy single-slot skydome picker ([`src/main.cpp:4915`](src/main.cpp:4915)) stays as-is. Only the engine-side data model (shared by both) changes; legacy keeps showing one slot.
- **Authoring/saving skydome selection back into a `.ted` map.** MT-15 reads environment data; it does not write maps. Separate concern, no ROADMAP entry requests it.
- **A general-purpose static `.alo` mesh loader/renderer as a shipped feature** beyond what the skydome needs. If mesh rendering proves required (¬ß3a), scope it to the skydome dome only; a general model viewer is out.

**LT-8 coupling (design constraint, not deferred work).** LT-8 needs per-map colour-grade params, which come from the **same map-environment source** MT-15 must locate. Build the "where does the active game/mod define environment data, and how do we read it" layer as a small reusable reader (a struct + a `FileManager`-backed loader ‚ÄĒ concrete shape proposed in ¬ß3, "Reusable reader contract"), **not** inlined into the skydome texture path. If MT-15 hardcodes skydome resolution the way the current curated table does ([`src/engine.cpp:55`](src/engine.cpp:55)), LT-8 pays the discovery cost again.

### 2.4 Mesh-loading status ‚ÄĒ the central gap

- **The game's real skydome is a textured `.alo` MESH**, not an equirectangular sphere. First-party evidence: [`reference/foc-shaders/Skydome.fx`](reference/foc-shaders/Skydome.fx) declares `_ALAMO_VERTEX_TYPE="alD3dVertNU2C"` (pos+normal+2 UV sets+colour, [`:26`](reference/foc-shaders/Skydome.fx:26)), `_ALAMO_RENDER_PHASE="Opaque"` ([`:25`](reference/foc-shaders/Skydome.fx:25)), transforms by `m_worldViewProj`/`m_world` (authored geometry), and renders **BaseTexture** (opaque background, [`:55-59`](reference/foc-shaders/Skydome.fx:55)) + **CloudTexture** (alpha-blended scrolling overlay; `CloudScrollRate` [`:39`](reference/foc-shaders/Skydome.fx:39), `CloudScale` [`:47`](reference/foc-shaders/Skydome.fx:47), scroll/scale applied in VS [`:136-137`](reference/foc-shaders/Skydome.fx:136), blended `lerp(...cloudTexel.a * In.Diff.a)` [`:161`](reference/foc-shaders/Skydome.fx:161)) + per-vertex SH diffuse `Sph_Compute_Diffuse_Light_All` ([`:144`](reference/foc-shaders/Skydome.fx:144); real order-2 quadratic body verified in [`reference/foc-shaders/AlamoEngine.fxh:173-181`](reference/foc-shaders/AlamoEngine.fxh:173)) + Emissive ([`:37`](reference/foc-shaders/Skydome.fx:37)). Shader targets `vs_1_1`/`ps_1_1` ([`:169-170`](reference/foc-shaders/Skydome.fx:169)).
  - **Cloud scroll is time-driven.** `CloudScrollRate` is applied in the VS to animate the cloud UVs ([`:136-137`](reference/foc-shaders/Skydome.fx:136)) ‚ÄĒ i.e. the real dome needs a **per-frame time uniform**. The editor has a single canonical sim clock: `GetTimeF()` funnels "simulation now" for every consumer (emitter spawn, particle dt, the shader `hTime` uniform, spawner dt ‚ÄĒ [`src/engine.cpp:114-119`](src/engine.cpp:114)). But the **dome path does not consume it**: `InitSkydomeEffect` caches only `g_WorldViewProj` + `g_Skydome` (¬ß2.1), and `RenderSkydome` sets no time/scroll handle. So animated clouds in MT-15 are **not free** ‚ÄĒ they require a new time handle wired through `InitSkydomeEffect` (cache it) and `RenderSkydome` (`SetFloat` from `GetTimeF()` each frame). See fork (a) sub-fork.
  - **Correction worth recording:** the header advertises atmosphere **fog**, but fog is **inert** in this variant ‚ÄĒ `Out.Fog = 1.0f; //Compute_Fog(...)` is commented out ([`:150`](reference/foc-shaders/Skydome.fx:150)) and the technique sets `FogEnable=false`. Do not budget for fog parity.
- **The editor's `Skydome.fx` is a deliberate fork/simplification** ([`src/Resources/Engine/Skydome.fx:1-52`](src/Resources/Engine/Skydome.fx:1)): one texture `g_Skydome`, PS is pure `tex2D` ([`:42`](src/Resources/Engine/Skydome.fx:42)), **no clouds, no SH lighting** (flat-lit); VS forces depth just inside the far plane `o.Position.z = o.Position.w * 0.9999` ([`:35`](src/Resources/Engine/Skydome.fx:35)); `vs_2_0`/`ps_2_0`, one technique "Skydome", one pass. Sampling `AddressU=WRAP` / `AddressV=CLAMP` ([`:13-14`](src/Resources/Engine/Skydome.fx:13)) ‚ÄĒ **note this differs from the real shader's `AddressV=WRAP`** ([`reference/foc-shaders/Skydome.fx:79`](reference/foc-shaders/Skydome.fx:79)), a parity detail if MT-15 keeps a textured sphere.
  - **SH-lighting parity is a known-visible regression class, not a cosmetic detail.** The editor fork drops `Sph_Compute_Diffuse_Light_All` entirely; the dome renders pure-`tex2D` flat. This is the *same* 1:1 tension MT-14 just resolved for the ground ‚ÄĒ MT-14 shipped specifically to make an inert surface "respond to lighting" (MEMORY `project_mt14_ground_lighting`). An unlit dome where the game's dome is SH-lit is therefore a precedented, user-visible 1:1 violation, not an acceptable approximation by default. Whether it *materially* matters for a dome (vs the ground) is unverifiable here and is a ¬ß5a check (does the dome's SH diffuse visibly change appearance vs flat-lit?).
- **No static-mesh `.alo` loader/renderer exists anywhere.** The repo's `.alo` parser ([`src/ParticleSystem.cpp:318-397`](src/ParticleSystem.cpp:318), Emitter ctor [`:484-523`](src/ParticleSystem.cpp:484)) only understands the **particle-system** chunk dialect (`0x0001/0x0002/0x0003/0x0016/0x0029/0x1100/0x1101`). Grep for mesh-chunk ids, `MeshChunk`, `StaticModel`, `loadMesh`, `D3DXLoadMeshFromX`, `ID3DXMesh` over `src/` returns **no real hits**. Rendering a textured static mesh from a `.alo` is **net-new**. The `.alo` open dialog already defaults to `Data\Art\Models` ([`src/host/BridgeDispatcher.cpp:2118`](src/host/BridgeDispatcher.cpp:2118)) but never parses models there.

### 2.5 Reusable external reference: the MIT-licensed `alo-viewer` (retires most of the ¬ß2.4 risk)

The ¬ß2.4 "net-new + undocumented format" framing assumed we'd start from nothing. We don't. **[`GlyphXTools/alo-viewer`](https://github.com/GlyphXTools/alo-viewer)** is a complete, open-source ALO model viewer that parses *and* renders Alamo `.alo` models, and it is unusually well-matched to this codebase. _(Verified 2026-06-13 against the live repo via `gh api`; not yet vendored locally ‚ÄĒ a real port must re-read the current source.)_

- **License: MIT** (¬© 2017 GlyphX Tools) ‚ÄĒ **permissive**: we may port/adapt the source into this project with attribution. This removes the licensing blocker that a GPL reference would have imposed. _(Confirm the LICENSE text and add the attribution line when vendoring.)_
- **Language: C++** (842 KB C++ / 377 KB C / 5 KB HLSL) on a **DirectX 9** render engine ‚ÄĒ the **same API** the editor uses (D3D9Ex). The render path, not just the parser, is relevant.
- **Dependency surface is minimal and already aligned with this repo:** the only git submodule is `libs/dx9` = [`hrydgard/minidx9`](https://github.com/hrydgard/minidx9) (standalone DX9 headers; the editor already builds against D3DX9), plus **`libs/expat-2.2.0` ‚ÄĒ the exact expat version this editor already vendors** (`libs/expat-2.2.0`, seen compiling in the host MSBuild). So a port drags in essentially nothing new.
- **Directly relevant files** (paths in the alo-viewer repo):
  - `src/Assets/ChunkFile.{cpp,h}` ‚ÄĒ its Alamo chunk reader. The editor already has its **own** `ChunkReader`/`ChunkWriter` ([`src/ChunkFile.h:9`](src/ChunkFile.h:9)); both target the same chunk substrate, so the port is "adapt to our `ChunkReader`," not "import theirs."
  - `src/Assets/Models.{cpp,h}` ‚ÄĒ the model loader. `Models.h` exposes `Model` with `Bone` / `Mesh` / `SubMesh` and **`Mesh* ReadMesh(ChunkReader&)`, `ReadSubMesh(ChunkReader&, ‚Ä¶)`, `ReadBone(‚Ä¶)`** ‚ÄĒ i.e. the mesh/submesh chunk decode we need, already written against a `ChunkReader` interface. A static skydome needs only the **mesh + material/texture** subset (ignore bones/animation/proxies/dazzles/lights).
  - `src/RenderEngine/DirectX9/VertexFormats.{cpp,h}` ‚ÄĒ enumerates the **full Alamo vertex-format vocabulary** with byte layouts, including **`VERTEX_MESH_NU2C` = `Position + Normal + D3DCOLOR + TexCoord`** ‚ÄĒ exactly the skydome's `alD3dVertNU2C` (¬ß2.4). The "reverse-engineer the vertex layout from a real asset" ¬ß5a item is largely answered by this file.
  - `src/RenderEngine/DirectX9/VertexManager.{cpp,h}` ‚ÄĒ DX9 VB/IB management for those formats.
- **Net effect on fork (a):** route **a2 (faithful real mesh)** drops from "open-ended reverse-engineering spike" to "**port a focused static-mesh subset of a known-good MIT C++/DX9 reference** that already speaks our ChunkReader + expat + DX9." That makes a2 viable within a sane budget and weakens a1's main advantage (cheapness) ‚ÄĒ see the revised fork (a) below. **Caveat:** this is a *reference/port-source* claim verified at the repo-structure level (files, license, deps, the NU2C struct); the precise mesh/material **chunk IDs** and the exact `ReadMesh` body still need a real read of `Models.cpp` (and validation against a real `.alo`) during implementation ‚ÄĒ moved to the ¬ß5a checklist, but now as "map an existing reader," not "invent one."

## 3. Architecture / approach (with the open design forks)

**Recommended overall shape:** follow the ROADMAP's **phased split** (which the ROADMAP only floats parenthetically ‚ÄĒ this doc adopts it), and build a thin reusable map-environment reader rather than another hardcoded path table.

- **Phase A ‚ÄĒ load + render ONE real skydome.** A `SkydomeEnvironment` reader resolves a skydome reference (name/path) for the active game/mod via `FileManager`, loads its base (and, if cheap, cloud) texture via `LoadTextureViaFileManager`, and renders it through the existing dome scaffolding (¬ß2.1). No dual-slot, no new UI beyond a single "use game skydome" affordance. Proves the resolution + render path end-to-end against a real install.
- **Phase B ‚ÄĒ primary + secondary + dual-slot picker.** Add the second active slot/dome, the compose step, the dual-slot picker UI, schema/DTO fields, and persistence.

This keeps the biggest unknown (mesh rendering, fork 3a) isolated to Phase A, where it can be de-risked before any UI/dual-slot work is committed.

### Reusable reader contract (the LT-8 keystone ‚ÄĒ concrete proposal)

The map-environment reader is named as the architectural keystone (¬ß1 In, LT-8 coupling, Phase A, ¬ß4.6) but needs a concrete shape so LT-8's reuse contract is *specified*, not just promised. Proposed (Phase A creates the file; **all field/signature details below are a starting shape to confirm against a real mod at ¬ß5a, not verified facts**):

- **New file `src/SkydomeEnvironment.h`** (header-only struct + free-function loader, mirroring the lightweight style of existing readers; the *parser* implementation lives in a `.cpp` if it grows).
- **Struct** (proposed):
  ```cpp
  struct SkydomeRef {
      std::string gameObjectName;   // resolved skydome GameObject (tag TBD ¬ß5a)
      std::string baseTexturePath;  // resolves via FileManager::getFile
      std::string cloudTexturePath; // optional; "" if none
      std::string meshPath;         // .alo model path, "" if a1/sphere path
      float       cloudScrollRate;  // 0 = static / no anim (fork a sub-fork)
      float       cloudScale;
  };
  struct MapEnvironment {
      SkydomeRef primary;           // Dome1
      SkydomeRef secondary;         // Dome2 (may be empty / constrained ‚ÄĒ ¬ß5a fork b)
      bool       hasPrimary, hasSecondary;
      // LT-8 extension point ‚ÄĒ colour-grade params added here, NOT a new locator:
      // ColourGradeParams colourGrade;  // (LT-8)
  };
  ```
- **Loader** (proposed signature):
  ```cpp
  // Resolves the active game/mod's environment via the injected FileManager
  // (mod‚Üíbase‚ÜíMEG). Returns false + leaves `out` zeroed on total miss
  // (graceful-degrade to Off). Never throws; bounded like ChunkReader.
  bool LoadMapEnvironment(IFileManager& fm, MapEnvironment& out);
  ```
  Takes `IFileManager&` (the same injected interface the engine holds, ¬ß2.2) so it is testable with a mock FM and reusable by LT-8 unchanged. The **skydome-specific fields are separable from the file-location/parse logic** so LT-8 adds `colourGrade` fields rather than a second locator. Engine hooks this from the same `ReloadTextures()` re-resolve point (¬ß2.2) so mod-switch is honoured.

### Fork (a) ‚ÄĒ textured MESH vs texture-on-a-sphere; is static-mesh rendering net-new?
The game skydome is a `.alo` mesh (¬ß2.4, **high** confidence from `reference/foc-shaders/Skydome.fx`). The editor has **zero** mesh-chunk parsing or static-model rendering (**high** confidence). Two routes:
- **(a1) Approximate** ‚ÄĒ keep the procedural sphere, swap in the game's **base+cloud textures**, optionally compile the real `Skydome.fx`. Cheap; reuses everything in ¬ß2.1. But it is not the actual dome geometry, **drops the SH per-vertex diffuse lighting**, and sits in tension with the no-shader-fork / 1:1 rule (MEMORY `feedback_no_shader_fork_1to1_rendering`). The 1:1 tension applies on **two axes**, both of which a1 currently fails: **geometry** (sphere vs authored mesh) *and* **lighting** (flat-lit vs SH diffuse) ‚ÄĒ the latter is the exact regression class MT-14 shipped to fix for the ground (¬ß2.4).
  - **Sub-fork (a-clouds) ‚ÄĒ static overlay vs. scrolling.** Default: **static** cloud overlay (or none) ‚ÄĒ no time uniform, nothing new in the lifecycle. If scrolling is wanted, the dome effect needs a **new time handle** cached in `InitSkydomeEffect` and set each frame in `RenderSkydome` from `GetTimeF()` ([`src/engine.cpp:114-119`](src/engine.cpp:114)) ‚ÄĒ plus the blend-state save/restore tightening in ¬ß2.1. Decide at design time; if not chosen, scrolling clouds are scoped **Out** (¬ß1).
- **(a2) Faithful** ‚ÄĒ load the real `.alo` mesh + material and render it, recovering both authored geometry *and* SH lighting. **No longer greenfield (¬ß2.5):** port a focused static-mesh subset of the MIT-licensed `alo-viewer` (`ChunkFile` ‚Üí our `ChunkReader`, the `ReadMesh`/`ReadSubMesh` decode, `VERTEX_MESH_NU2C`/`VertexManager` for the DX9 buffers), then either run the real `Skydome.fx` or feed the mesh through a faithful effect. The chunk vocabulary is **documented in working C++**, not reverse-engineered; the dependency surface (ChunkReader + expat 2.2.0 + DX9) is already in this repo. Scope is now a **bounded port**, plausibly within the 12‚Äď20h envelope rather than blowing past it ‚ÄĒ though still the larger of the two routes.
- **Recommendation (medium confidence) ‚ÄĒ a2 is now the more attractive target, but Phase A still de-risks via a1 first.** The ¬ß2.5 reference removes a2's old cost penalty and its 1:1 advantage is real (it fixes *both* the geometry and SH-lighting axes a1 violates). However, the prudent sequence is unchanged in shape: **Phase A ships a1** (textured sphere + real game textures) to prove the resolution + render plumbing and deliver a usable background fast, **then port a2** ‚ÄĒ either as the Phase-A render leaf if the ¬ß5a materiality check comes back "SH/geometry visibly matter" (likely, given MT-14's precedent), or as a fast-follow. Because a2 is now a bounded port, **lean toward scheduling it rather than treating it as a stretch goal.** The a1-vs-a2 *fidelity* judgement still **blocks on real-install confirmation** (¬ß5a) ‚ÄĒ you can't see whether the real dome's mesh/SH actually differ from a flat projection without one ‚ÄĒ but the *feasibility* of a2 is no longer in doubt.

### Fork (b) ‚ÄĒ how do primary + secondary compose?
`reference/foc-shaders/Skydome.fx` describes **one** dome's base+cloud and says nothing about a second dome (**high** that the shader is single-dome; **low** on the two-dome layering, community-only). Community sources consistently report Dome1=starfield (primary), Dome2=backdrop (secondary), implying two concentric domes drawn back-to-front. Candidates: (i) two depth-layered opaque domes; (ii) secondary alpha-blended over primary; (iii) base+cloud of a single dome mistaken for "two slots."
- **Independence is itself unconfirmed.** Whether primary and secondary are *independently* free-chosen from one enumerated list (the ¬ß1 goal / fork e UI assumption) or whether selecting one **implies/constrains** the other (e.g. a dome GameObject references its own paired secondary) is **low** confidence. If the game pairs them, a dual-free-picker UI (fork e) is wrong. This is a ¬ß5a gate that fork (e)'s UI shape depends on.
- **Recommendation (low confidence):** assume **two independent domes drawn back-to-front**, secondary first; design `RenderSkydome` to take a (texture, optional blend) pair and draw twice.
  - **Dual-dome depth/blend correctness (see ¬ß4.7).** With `ZWRITE`/`ZENABLE` forced OFF (¬ß2.1), there is **no depth arbitration between the two domes** ‚ÄĒ layering is purely **draw order**. The second dome drawn always paints over the first in the overlapping region. So: draw the **primary first**, secondary second only if the secondary is meant to sit in front; if both are opaque full domes, the second fully occludes the first (a bug unless intended). An alpha-blended secondary (candidate b-ii) needs the **full blend delta set** added to the save/restore block ‚ÄĒ `ALPHABLENDENABLE` + `SRCBLEND` + `DESTBLEND` with **explicit factors** (not whatever the prior pass left), per ¬ß2.1's L-032 warning ‚ÄĒ and correct src/dst factors are an open render-state question, not given.
- **BLOCKS on real-install confirmation** (load a map with both domes set and observe; confirm independence and compose model).

### Fork (c) ‚ÄĒ where are skydomes referenced?
Nothing in-repo reads any environment/map file (**high** ‚ÄĒ verified absent). Community sources (**low**): skydomes are XML GameObjects registered via `GameObjectFiles.xml`, the chosen domes stored as references in the binary `.ted` map. Candidate sources: per-map `.ted` environment chunk, a global/mod default, or a curated list.
- **Recommendation (medium confidence):** for the **picker list**, enumerate skydome GameObjects from the XML registry (reuse the proven `XMLTree` parser, ¬ß2.2) rather than reading a binary `.ted` (which needs a map open and a reverse-engineered chunk id). Treat the `.ted` per-map selection as a **later** input (and an LT-8 concern). Build the reader as the reusable `SkydomeEnvironment`/`MapEnvironment` layer (signature above). **BLOCKS on real-install confirmation** ‚ÄĒ the exact GameObject XML tag and which file enumerates the available domes are unverifiable here.

### Fork (d) ‚ÄĒ keep or retire the texture picker?
ROADMAP defers this ([`ROADMAP.md:64-66`](ROADMAP.md:64) ‚ÄĒ "decide during design"); this doc decides it. The current 12-slot model is the only quick neutral-background path.
- **Recommendation (high confidence):** **keep** solid-colour (slot 0) + custom slots as a "simple background" fallback; **demote** the 8 fake-gradient bundled scenes (they were never real assets, ¬ß2.3) and add the real game-skydome picker as the primary mode. Surgical, preserves the neutral-background workflow, removes the only misleading part.
- **Empty-state UX (depends on ¬ß5a base-game-only check).** If a stock FoC install (no mod) exposes skydome GameObjects, the game-skydome mode populates from base game and the picker is functional with no mod selected. **If base game exposes none ‚ÄĒ or there is no game install at all ‚ÄĒ the game-skydome mode is empty.** Intended empty-state: the game-skydome section shows a disabled/empty list with an explanatory hint ("Select a mod to choose a skydome" or "No FoC install detected"), while the **simple-background fallback (solid colour + custom paths) remains fully functional** ‚ÄĒ i.e. the editor is never left with *no* background option. This mirrors the native-harness story (¬ß5b): with no install, only bundled-RCDATA fallback + custom paths work. The exact base-game enumeration behaviour is a ¬ß5a gate.

### Fork (e) ‚ÄĒ dual-slot picker UI placement
- **Recommendation (high confidence):** replace `BackgroundPickerBody`'s content inside the existing `BackgroundDropdown` popover ([`Toolbar.tsx:229`](web/apps/editor/src/components/Toolbar.tsx:229)) with a picker exposing primary + secondary. It is already extracted for reuse ([`:77`](web/apps/editor/src/screens/BackgroundPicker.tsx:77)).
- **UI shape is GATED on fork (b) independence (¬ß5a).** If primary/secondary are independently selectable, build **two free-choice sub-pickers** (one popover with two sections, vs two separate toolbar dropdowns mirroring the map editor ‚ÄĒ sub-question resolved after seeing the map editor, **low** confidence). If selecting one **constrains** the other, the UI is a single primary picker with the secondary shown as derived/locked, not a second free chooser. Do not build two free pickers until ¬ß5a confirms independence.
- Also gate the **game-skydome section's empty state** here on fork (d)/¬ß5a (no mod / no install ‚Üí disabled list + simple-background fallback stays live).

## 4. Risks + mitigations

1. **Mesh rendering is net-new ‚ÄĒ but no longer an open-ended unknown (¬ß2.5).** Hazard: if real-install inspection shows the dome geometry *or* its SH lighting materially differs from a flat-lit equirectangular projection, route a1 fails fidelity and the team must add an `.alo` mesh load + static-mesh draw (and recover SH) ‚ÄĒ which is still real work the editor has never done. Mitigation (now much stronger): the MIT-licensed `alo-viewer` (¬ß2.5) is a **port source** for exactly this ‚ÄĒ its `Models.cpp` `ReadMesh`/`ReadSubMesh` (already written against a `ChunkReader`) and `VertexFormats` (`VERTEX_MESH_NU2C`) turn the work into a bounded port, not a research spike, and it shares this repo's deps (ChunkReader + expat 2.2.0 + DX9). Sequence: build the `SkydomeEnvironment`/`MapEnvironment` reader (¬ß3) + a1 texture path first so a2, if/when taken, only swaps the *render* leaf, not the resolution layer; reuse the editor's `ChunkReader` ([`src/ChunkFile.h:9`](src/ChunkFile.h:9)) and `ReadAndRelease` ([`src/files.cpp:10`](src/files.cpp:10)) as the local base the ported decode targets. The residual risk is integration effort + the ¬ß5a materiality check, not feasibility.
2. **The asset format is unconfirmable on this dev box.** Hazard: the GameObject XML tag, the dome-list source file, the `.ted` reference chunk, primary/secondary independence, and the dome compose model are all **low**-confidence community knowledge; designing against them risks building the wrong reader. Mitigation: ¬ß5a is a hard gate ‚ÄĒ **no data-model code until the format is confirmed against a real install/mod**; until then the reader is an interface (the ¬ß3 signature), not an implementation.
3. **Mod/base fallback edge cases (incl. asymmetric primary/secondary miss).** Hazard: a mod may override only the base texture, or reference a dome that resolves in base-game but whose texture is mod-only, or be missing entirely; with two domes, **one may resolve while the other misses**. `getFile` already handles mod‚Üíbase‚ÜíMEG ([`src/managers.cpp:13`](src/managers.cpp:13)) and returns `NULL` on total miss, but the **un-pooled** `D3DXCreateTextureFromFileInMemory` fallback path ([`src/engine.cpp:2313`](src/engine.cpp:2313)) relies on `D3DPOOL_MANAGED`, **invalid under D3D9Ex** ‚ÄĒ it is only safe today because the FileManager branch returns first when assets exist. Mitigation: any new texture-load path must pass `D3DPOOL_DEFAULT` explicitly (as the custom-slot path does, [`src/engine.cpp:2336`](src/engine.cpp:2336)); hook the dome re-resolve into `ReloadTextures()` ([`src/engine.cpp:613`](src/engine.cpp:613)) so mod-switch is honoured; handle every resolution miss with graceful-degrade (mirror the slot-set fallback at [`src/engine.cpp:2599`](src/engine.cpp:2599)) ‚ÄĒ and critically, **degrade per-slot**: a missing secondary must not disable a resolved primary (and vice versa). See ¬ß5b asymmetric-miss test.
4. **New-UI persistence gap.** Hazard: dual-slot selection set via the bridge is **not persisted** (handlers only `markDirty`, ¬ß2.3) ‚ÄĒ it silently resets every restart in the daily-driver. Mitigation: extend the persistence seam ‚ÄĒ either add registry write-back to the bridge setters (mirror `settings/lighting-force-align/set` at [`src/host/BridgeDispatcher.cpp:1755`](src/host/BridgeDispatcher.cpp:1755)) or define new `Skydome{Primary,Secondary}` keys read by [`src/host/HostWindow.cpp:2038`](src/host/HostWindow.cpp:2038). Decide which in Phase B.
5. **Legacy-path scope creep.** Hazard: the shared slot constants ([`src/engine.h:332`](src/engine.h:332)) and shared engine API mean a naive dual-slot change can break the legacy single-slot picker ([`src/main.cpp:4915`](src/main.cpp:4915)). Mitigation: keep the existing single-`m_skydomeIndex` API working (legacy keeps one slot per scope-Out ¬ß1); add the secondary as **additive** engine state, not a replacement of the slot scalar.
6. **LT-8 reusability debt.** Hazard: if the map-environment reader is built narrowly (skydome-only, or a second hardcoded path table √† la [`src/engine.cpp:55`](src/engine.cpp:55)), LT-8 re-pays the discovery cost for per-map colour-grade. Mitigation: build the `SkydomeEnvironment`/`MapEnvironment` reader (concrete shape in ¬ß3) that returns a struct, with the file-location/parse logic separable from the skydome-specific fields, so LT-8 adds colour-grade fields rather than a new locator.
7. **Dual-dome depth/blend correctness (L-032 class, two domes).** Hazard: `RenderSkydome` forces `ZWRITE FALSE` / `ZENABLE FALSE` and the editor VS pushes z to `w*0.9999` (¬ß2.1/¬ß2.4), so two domes have **no depth arbitration** ‚ÄĒ the second drawn overpaints the first regardless of intended layering. An opaque secondary over an opaque primary fully occludes it (over-paint bug); an alpha secondary needs **explicit `SRCBLEND`/`DESTBLEND` factors** plus `ALPHABLENDENABLE`, all added to the save/restore block (under-specifying that set repeats L-032). Mitigation: rely on **draw order alone** for layering ‚ÄĒ primary first, secondary second; set explicit blend factors for any alpha secondary; **save and restore the full delta state set** (`ALPHABLENDENABLE` + `SRCBLEND` + `DESTBLEND` + any second-texture sampler/stage state ‚ÄĒ ¬ß2.1); verify no over-paint when both are opaque full domes and no z-fighting (there can't be, with z off ‚ÄĒ confirm the visual is the intended back-to-front, not an accidental occlusion). Also verify particles still don't whiten (L-032, [`tasks/lessons.md:2563`](tasks/lessons.md:2563)) after the now-larger save/restore block.
